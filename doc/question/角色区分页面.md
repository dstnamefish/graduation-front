### 一、 数据库层面的深度优化（针对业务闭环）

目前你的数据库（基于 `V1.0__Initial_Schema.sql`）已有用户、角色、医生、患者、预约等核心表。但要支撑复杂的仪表盘和角色流转，建议新增/优化以下 5 个关键部分：

| 优化方向 | 建议新增/修改的表格 | 设计意图 |
| :--- | :--- | :--- |
| **系统通信** | `sys_announcement` (系统公告表) | 支撑首页仪表盘顶部的“轮播公告”或“紧急通知”。 |
| **运营快照** | `stat_daily_snapshot` (每日运营统计表) | 仪表盘如果直接查 `invoice` (发票) 会很慢。每天凌晨统计一次营收、门诊量存入此表，首页秒开。 |
| **排班增强** | 修改 `doctor_schedule` 增加 `max_appointment` | 限制每个班次的预约上限，防止医生过载，并在仪表盘显示“号源饱和度”。 |
| **评价闭环** | 修改 `feedback` 关联 `appointment_id` | 将评价与具体的就诊记录挂钩，科室管理员可以追踪到是哪一次服务出了问题。 |
| **审计追溯** | 修改 `sys_audit_log` 增加 `impact_level` | 区分普通操作和“高危操作”（如大额退费、删档），超级管理员仪表盘优先推送高危日志。 |

### 二、 角色页面映射与功能归属（基于现有 + 补齐）

根据你的 `src/views/` 目录，我将页面划分为三个核心角色视角。**红色字体**代表建议你新增加的页面，以填补业务空白。

#### 1. 超级管理员 (SUPER_ADMIN) —— “全院指挥官”
* **核心目标**：监控、决策、全局配置。
* **现有页面归属**：
    * `dashboard/index.vue` (加载超级大屏版)
    * `department/list/index.vue` (全院科室增删改)
    * `doctor/list/index.vue` (全院医生档案库)
    * `payment/index.vue` (全院财务流水与对账)
    * `inventory/index.vue` (全院药库物资总览)
* **建议新增页面**：
    * **`system/user/index.vue`** (系统账号与权限分配)
    * **`system/audit/index.vue`** (安全审计与操作日志查询)
    * **`system/announcement/index.vue`** (发布全院/全系统公告)

#### 2. 科室管理员 (DEPARTMENT_ADMIN) —— “运营调度中心”
* **核心目标**：人力分配、流程监控、投诉处理。
* **现有页面归属**：
    * `dashboard/index.vue` (加载科室水位版)
    * `doctor/schedule/index.vue` (**最核心：** 负责本科室医生排班的增删改)
    * `appointment/index.vue` (监控本科室所有预约，处理强制退号/加号)
* **建议新增页面**：
    * **`feedback/management/index.vue`** (处理针对本科室医生的患者投诉与评价)
    * **`department/performance/index.vue`** (本科室医生接诊量、营收贡献排名)

#### 3. 医生 (DOCTOR) —— “沉浸式诊疗工作站”
* **核心目标**：高效接诊、病历书写。
* **现有页面归属**：
    * `dashboard/index.vue` (加载医生待办版)
    * `patient/list/index.vue` (查看自己接诊过的患者)
    * `patient/detail/index.vue` (**最核心：** 书写病历、开处方、查报告)
    * `message/index.vue` (与同事沟通，接收科室通知)
* **建议新增页面**：
    * **`doctor/my-schedule/index.vue`** (个人出诊日历，支持一键发起调班申请)

### 四、 如何在代码中实现这些重组？

不要为每个角色复制页面，而是利用你的 `src/hooks/core/useAuth.ts` 进行逻辑拆分。

**1. 路由过滤 (`router/index.ts`)**：
在 `MenuProcessor.ts` 中根据后端的 `role_id` 过滤掉不属于该角色的菜单入口。

**2. 仪表盘动态拼装 (`dashboard/index.vue`)**：
```typescript
// 伪代码思路
const renderLayout = computed(() => {
  if (role === 'SUPER_ADMIN') return AdminLayout; // 包含营收、审计模块
  if (role === 'DEPT_ADMIN') return DeptLayout;   // 包含排班、反馈模块
  return DoctorLayout;                            // 包含待诊、日程模块
});
```

**3. 按钮级权限控制 (`Wn-button`)**：
给 `Wn-button` 增加一个指令或属性，例如 `<Wn-button v-if="hasAuth('DELETE_DOCTOR')">`。

### 五、 产品经理的建议：你还缺什么？

为了让你的毕业设计/项目展示更完美，我建议再增加 **1个隐藏模块**：

* **药房/发药工作站 (`views/pharmacy/index.vue`)**
    * **原因**：你已经有了“开处方（Prescription）”和“收钱（Invoice）”，但没有“发药”。
    * **流程**：医生开了药 -> 患者交了钱 -> 药房看到“待发药”状态 -> 点击发药 -> **扣减 `inventory` 表的库存**。
    * **价值**：这一步实现了数据库的物理更新闭环，是评审最看重的逻辑完整性。

这套方案不仅能让你的代码量和表格深度增加，还能让整个系统的 UI 呈现出一种**“各司其职、互不干扰”**的专业 SaaS 质感。需要我帮你针对其中某一个新增表格（如发药站）写出 SQL 或 Vue 模板吗？
